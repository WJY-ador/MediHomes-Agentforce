## 논의 사항

### 1. 에이전트포스 활용 방안 — Phase 4단계 제안 (오예희 AE)

메디홈즈 측이 제시한 4가지 요구사항을 Phase 구조로 재구성. 고객이 가장 빨리 체감하는 ROI부터 순차 확장 방식.

- **Phase 1 — FAQ 기반 초기 수면 케어**
  - 양압기 사용 최대 고비는 초반 3개월 (이혜정 이사 언급)
  - 공유받은 FAQ 시트 + 현재 카테고리 구조 활용
  - 단순 FAQ 검색이 아닌 **증상(Symptom) 베이스의 가이드 플로우**로 제공
- **Phase 2 — 개인화(Personalization) 정보 조회**
  - "콜센터로 전화하지 않아도 알 수 있는 나의 정보 조회"
  - 고객별 인증 링크 → Salesforce patient record 조회 → Self-Service로 응답
- **Phase 3 — 기기 진단 및 구매 연결**
  - 소모품 교체 주기 문의 시 구매 링크/교체 주기 안내
  - 단순 응답이 아닌 **Next Action**(구매 여정) 연결 흐름이 핵심
- **Phase 4 — 수면 질 향상 컨텐츠 (Proactive Care)**
  - 숙면 환경 가이드, 여행 시 주의사항 등
  - 메디홈즈 궁극 비전: 환자를 Proactive하게 케어하는 수면 플랫폼
  - 고객이 묻기 전에 Push 기반으로 먼저 케어

### 2. 요구사항 재분류 및 Phase 4 축소 방향 (이재승 대표)

에이전트포스만으로 모든 요구를 커버할 수 없어 혼재된 요구사항을 분리해야 한다는 지적.

- **4가지 영역 구분**
  - (1) 보편적 FAQ/지식문서로 대응 가능한 영역
  - (2) 개인 사용자 식별 후 맞춤 대응 영역
  - (3) 케이스/신규 리드로 자동 인입되는 채널 앞단 **클린징 역할**
  - (4) Proactive Push 영역은 현시점 방향성이 또렷하지 않음
- **메시지 전략**: "현재 주어진 데이터/시나리오만으로 충분히 서비스 확장이 가능하다"는 점을 현장에서 전달. 계약 이후 아이디어 추가 가능
- **Phase 4 결론**: Push 기반 아웃바운드는 **Marketing Cloud 영역** → 이번 도입 범위 아님. 안정화 이후 마케팅 클라우드 슈거로 계약 후 캠페인 설계 필요
  - 단, 에이전트포스 **대화 말미에 추천(Recommendation) 멘트** 형태로 붙이는 방식은 현재 범위에서 가능 → 이 방향으로 시나리오 설계

### 3. 시나리오별 데이터 매핑 (정호윤 매니저)

전달받은 `Medihomes_개체별 필드명.xlsx`의 처방전 중심 오브젝트에서 ERD를 추정하여 시나리오 커버 가능성 검토.

- 추정 ERD: 처방전 ← Account(요양기관) / 환자(고객) / 고객주치의 등
- **마스크 트러블**: 환자 → Order/Asset/Product 연결 (임의 구성 필요하나 가능)
- **수면 상태 확인**: 수면평가 시작일/종료일/하루 평균 사용시간 필드 존재 → "이번 달 몇 시간 더 사용해야 보험 대상" 역산 가능
- **가습 온도 설정**: FAQ + 모델별(에셋별) 설정 데이터 임의 구성
- **처방전 만료일/시작일/발행일**: 제공 필드만으로 조회 가능
- **본인 인증**: Account/환자 기반, SMS 연동 인증 방식으로 구상

### 4. 고객 인증 플로우 확정

카카오 싱크는 과도 → SMS/알림톡 인증코드 방식으로 합의. 오예희 본인이 다니는 병원 앱에서 동일한 플로우 실사용 사례 확인하여 검증됨.

**최종 인증 플로우**

1. 일반 문의는 비식별 상태로 응답
2. 개인 정보 조회가 필요한 지점에서 인증 모달 띄움
3. 고객에게 **이름 + 생년월일** 입력받음 (전화번호 직접 입력은 식별/인증이 아님)
4. 입력값으로 Account/환자 레코드 조회
5. 레코드에 등록된 전화번호로 **6자리 난수 인증번호** 발송
   - 유료: 카카오 알림톡
   - 무료: 이메일
6. 웹챗에서 입력된 코드와 발송 코드 대조 → 일치 시 식별 완료
7. 이후 환자 데이터 기반 응답 진행

### 5. 로그인 강제 여부

- 타사 사례: 비너스 = 100% 로그인 강제 / 오뚜기 = 부분 식별 고민 중
- 메디홈즈는 단순 질의 비중이 크다고 고객이 언급 → **전체 로그인 강제는 불필요**
- 하이브리드 방향: 앞단에서 단순 질의를 먼저 클린징 → 개인화 필요 시점에서만 인증 진행
- 로그인 강제 여부는 고객 전략(업셀 활용 의도 등)에 따라 **향후 결정 가능한 문제**로 정리

### 6. Data Cloud(D360) 필요 여부

- **데모 수준**: D360 없이도 오브젝트 레코드 조회 + Knowledge 액세스로 동일 결과 확보 가능 (최근 데모 선례 확인)
- **실제 구축/크레딧 산정 시점**: D360 필요 (RAG 형태 FAQ 적재, 멀티 데이터 소스 연동)
- **현장 전달 방식**: 데모 씬은 에이전트포스 라이브로, D360은 **아키텍처 다이어그램/스크린샷 + 구두 설명**으로 대체 가능
- 보람님(세일즈포스) 주도로 D360 포지셔닝 설명 / 크레딧 산정 플로우 제시

### 7. 데모 환경(SDO) 구성

- 메디홈즈 실제 환경 미접근 → 제공받은 데이터 기반 유추하여 뉴브릭이 구성
- SDO 포함 품목: **Agentforce, Data 360(D360), Digital Engagement**
- 뉴브릭 구성 예상 소요: **약 1주일**

### 8. 고객 아키텍처 다이어그램

- 메디홈즈 현 환경: 파트너 커뮤니티 + CRM + 웹사이트(작업 중) + 외주 업체 인입 채널 등
- 세일즈포스 **SE에게 전체 아키텍처 다이어그램 작성 요청** → 솔루션 포지셔닝 이해 확보 목적
- 뉴브릭과도 공유 예정

### 9. 견적 관련

- 1차 미팅에서 이미 러프 견적 전달: 라이센스 + 구축비 합산 **약 2~3억 규모**
- 데모 미팅 당일에는 100% 확정 견적은 어려움 → **대략적 수준**으로 준비
- 추가 지표가 필요할 경우 보람님이 현장에서 보완 (플랫크레딧 관련)
- 가급적 미팅 당일 의사결정까지 연결 시도

### 10. 규제 관련 참고 (여행 트래킹 사유)

- 한국 정부 의료기기 대여사업 규제상, 한 달 중 **10일 이상 해외 체류 시 청구 불가**
- 청구 금액 변동 사유로 고객의 해외 체류 여부 트래킹 필요 → 여행 관련 프로액티브 케어 아이디어의 실질적 근거
